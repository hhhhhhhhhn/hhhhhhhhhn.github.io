Understanding RSA

RSA is a beautifully simple algorithm for asymmetric encryption, which uses the properties of modular arithmetic to transmit small messages securely. This article presents a possible thought process from which the algorithm could be derived, based on some key mathematical facts.

Note: This article requires some knowledge of modular arithmetic.

A Small Overview

RSA is an asymmetric encryption algorithm, which means that its goal is to create a message which is encrypted using one key, but decrypted using a different one. In practice, it is a way to ensure that a message is only received by a specific person. To do this, the receiver of the messages creates a public encryption key, which any person can use to send them a message, and a decryption key, which they keep private and is the only way to read the messages sent.

The Key Insight

The starting point for the RSA algorithm is a simple fact derived from Euler's Theorem: If you have a congruence m\displaystyle{ m }m (for message) in a modulo n\displaystyle{ n }n , and they share no common factors, then there are infinitely many powers x\displaystyle{ x }x such that:

mx≡m (mod n)\displaystyle{ m ^{ x } ≡ m \text{ (mod n)} }mx≡m (mod n)

For instance:

125≡1 (mod 35)\displaystyle{ 1 ^{ 25 } ≡ 1 \text{ (mod 35)} }125≡1 (mod 35)
225≡2 (mod 35)\displaystyle{ 2 ^{ 25 } ≡ 2 \text{ (mod 35)} }225≡2 (mod 35)
325≡3 (mod 35)\displaystyle{ 3 ^{ 25 } ≡ 3 \text{ (mod 35)} }325≡3 (mod 35)
425≡4 (mod 35)\displaystyle{ 4 ^{ 25 } ≡ 4 \text{ (mod 35)} }425≡4 (mod 35)

Basically, the entire exponentiation becomes a no-op. This may not seem very useful, but if we rewrite x\displaystyle{ x }x as the product of two numbers, let's say, x=e⋅d\displaystyle{ x = e \cdot d }x=e⋅d , then, it must hold that:

me⋅d≡m (mod n)\displaystyle{ m ^{ e \cdot d } ≡ m \text{ (mod n)} }me⋅d≡m (mod n)
(me)d≡m (mod n)\displaystyle{ \left( m ^{ e } \right) ^{ d } ≡ m \text{ (mod n)} }(me)d≡m (mod n)

Which, if you look at it, is exactly what we want. Anyone with the encryption key e\displaystyle{ e }e can encode a message:

encrypted message≡me (mod n)\displaystyle{ \text{encrypted message} ≡ m ^{ e } \text{ (mod n)} }encrypted message≡me (mod n)

And, from the encrypted message, you can only recover the original one with the decryption key d\displaystyle{ d }d .

(encrypted message)d≡m (mod n)\displaystyle{ \left( \text{encrypted message} \right) ^{ d } ≡ m \text{ (mod n)} }(encrypted message)d≡m (mod n)

This is the base of how the encryption scheme works. We create public n\displaystyle{ n }n and e\displaystyle{ e }e values, which anyone can use to encrypt a message, and we, and only we, can use our private key d\displaystyle{ d }d to decrypt it. But now, we need to find a way of actually creating n\displaystyle{ n }n , e\displaystyle{ e }e and d\displaystyle{ d }d , such that the original property holds, and more importantly, such that it is not easily possible to derive the decryption key d\displaystyle{ d }d from the other variables, which would defeat the whole point.

Diving Deeper into Euler's Theorem

First of all, we must understand Euler's Theorem, the source of our key insight. It states that for a congruence m\displaystyle{ m }m in a modulo n\displaystyle{ n }n , with no common factors between m\displaystyle{ m }m and n\displaystyle{ n }n , it holds that:

mφ(n)≡1 (mod n)\displaystyle{ m ^{ φ \left( n \right) } ≡ 1 \text{ (mod n)} }mφ(n)≡1 (mod n)

Where φ(n)\displaystyle{ φ \left( n \right) }φ(n) is Euler's totient function, which requires finding the prime factors of n\displaystyle{ n }n , and is generally hard to compute.

This equation, in turn, means that multiplying anything by mφ(n)\displaystyle{ m ^{ φ \left( n \right) } }mφ(n) has no effect (as it is congruent to 1), and we can do it freely. From this, we get that:

m≡m⋅mφ(n)≡m⋅mφ(n)⋅mφ(n)≡m⋅mφ(n)⋅…⋅mφ(n)≡m1+qφ(n) (mod n)\displaystyle{ m ≡ m \cdot m ^{ φ \left( n \right) } ≡ m \cdot m ^{ φ \left( n \right) } \cdot m ^{ φ \left( n \right) } ≡ m \cdot m ^{ φ \left( n \right) } \cdot \ldots \cdot m ^{ φ \left( n \right) } ≡ m ^{ 1 + q φ \left( n \right) } \text{ (mod n)} }m≡m⋅mφ(n)≡m⋅mφ(n)⋅mφ(n)≡m⋅mφ(n)⋅…⋅mφ(n)≡m1+qφ(n) (mod n)
mqφ(n)+1≡m (mod n)\displaystyle{ m ^{ q φ \left( n \right) + 1 } ≡ m \text{ (mod n)} }mqφ(n)+1≡m (mod n)

Where q\displaystyle{ q }q is a whole number.

Now, if we set x=e⋅d=qφ(n)+1\displaystyle{ x = e \cdot d = q φ \left( n \right) + 1 }x=e⋅d=qφ(n)+1 , we have recreated the original property:

mx≡me⋅d≡m (mod n)\displaystyle{ m ^{ x } ≡ m ^{ e \cdot d } ≡ m \text{ (mod n)} }mx≡me⋅d≡m (mod n)

We have also greatly constrained the possible values of e\displaystyle{ e }e and d\displaystyle{ d }d , with e⋅d=qφ(n)+1\displaystyle{ e \cdot d = q φ \left( n \right) + 1 }e⋅d=qφ(n)+1 . This equation can be rewritten in a modular form:

e⋅d≡1 (mod φ(n))\displaystyle{ e \cdot d ≡ 1 \text{ (mod φ(n))} }e⋅d≡1 (mod φ(n))
d≡e−1 (mod φ(n))\displaystyle{ d ≡ e ^{ - 1 } \text{ (mod φ(n))} }d≡e−1 (mod φ(n))

This establishes a clear relationship between e\displaystyle{ e }e and d\displaystyle{ d }d , which can be further simplified by the fact that e\displaystyle{ e }e is supposed to be public, which allows us to use a preset value (usually the prime 216+1\displaystyle{ 2 ^{ 16 } + 1 }216+1 ) without worrying too much about security. We then only need to make sure e\displaystyle{ e }e is coprime with φ(n)\displaystyle{ φ \left( n \right) }φ(n) for d\displaystyle{ d }d to exist (made easier by the primality of e\displaystyle{ e }e ). We have also gained an important requirement: To compute d\displaystyle{ d }d , we basically only need to know φ(n)\displaystyle{ φ \left( n \right) }φ(n) . This means φ(n)\displaystyle{ φ \left( n \right) }φ(n) should be kept private as well. Then, finding d\displaystyle{ d }d from e\displaystyle{ e }e is just a multiplicative inverse, which can be done using the Extended Euclidean Algorithm.

Euler's Totient Function

To complete the algorithm, we need to actually compute φ(n)\displaystyle{ φ \left( n \right) }φ(n) . For our purposes, it necessary to know that the totient function is computed with all of n\displaystyle{ n }n 's unique prime factors p1,p2,…,pk\displaystyle{ p _{ 1 } , p _{ 2 } , \ldots , p _{ k } }p1​,p2​,…,pk​ :

Computationally, this means that finding φ(n)\displaystyle{ φ \left( n \right) }φ(n) requires factoring n\displaystyle{ n }n , which is a famously hard problem to do efficiently with large numbers. On one hand, this means that people will not be able to easily compute φ(n)\displaystyle{ φ \left( n \right) }φ(n) , and, by extension, d\displaystyle{ d }d , which is exactly what we want. On the other hand, this also means we cannot either, which makes the entire algorithm unworkable.

The solution is to backwards: Instead of starting with an n\displaystyle{ n }n and factoring it, we start with the prime factors and multiply them to compute n\displaystyle{ n }n . The simplest non-trivial case for this is when n\displaystyle{ n }n is a product of two primes n=p⋅q\displaystyle{ n = p \cdot q }n=p⋅q . Then,

φ(n)=φ(p⋅q)=p⋅q⋅p−1p⋅q−1q=(p−1)(q−1)\displaystyle{ φ \left( n \right) = φ \left( p \cdot q \right) = p \cdot q \cdot \frac{ p - 1 }{ p } \cdot \frac{ q - 1 }{ q } = \left( p - 1 \right) \left( q - 1 \right) }φ(n)=φ(p⋅q)=p⋅q⋅pp−1​⋅qq−1​=(p−1)(q−1)

Now, if we generate two primes, computing n\displaystyle{ n }n and φ(n)\displaystyle{ φ \left( n \right) }φ(n) is trivial, but, given n\displaystyle{ n }n , finding φ(n)\displaystyle{ φ \left( n \right) }φ(n) requires finding the factors p\displaystyle{ p }p and q\displaystyle{ q }q , which for large enough numbers, becomes computationally unfeasible. This is the last step to our puzzle, and we are ready to describe the algorithm in full.

Putting It All Together

The first step in our algorithm is to generate two primes, p\displaystyle{ p }p and q\displaystyle{ q }q , from which we can compute our n=p⋅q\displaystyle{ n = p \cdot q }n=p⋅q and φ(n)=(p−1)(q−1)\displaystyle{ φ \left( n \right) = \left( p - 1 \right) \left( q - 1 \right) }φ(n)=(p−1)(q−1) . We share n\displaystyle{ n }n with everyone, as part of our public key, while keeping the other values private.

We then choose a value for our e\displaystyle{ e }e , usually e=216+1\displaystyle{ e = 2 ^{ 16 } + 1 }e=216+1 , and share it as the final part of our public encryption key. Then, using our private φ(n)\displaystyle{ φ \left( n \right) }φ(n) , we compute the private decryption key d≡e−1 (mod φ(n))\displaystyle{ d ≡ e ^{ - 1 } \text{ (mod φ(n))} }d≡e−1 (mod φ(n)) . We have now finished the key generation, and it is time to send some messages.

Any person that knows both n\displaystyle{ n }n and e\displaystyle{ e }e can encrypt a message m\displaystyle{ m }m with a simple exponential:

encrypted message≡me (mod n)\displaystyle{ \text{encrypted message} ≡ m ^{ e } \text{ (mod n)} }encrypted message≡me (mod n)

This message cannot be decrypted by anyone, even the original sender, without the private decryption key d\displaystyle{ d }d . Since we know it, we can decrypt it just as easily as it was encrypted:

m≡(encrypted message)d (mod n)\displaystyle{ m ≡ \left( \text{encrypted message} \right) ^{ d } \text{ (mod n)} }m≡(encrypted message)d (mod n)

And thus, we have gotten back the original message, without the possibility of snooping by third parties.

Conclusion

In this article we have gone through the mathematical principles that allow the RSA algorithm to work, alongside a possible process for going from these insights to an actual algorithm. That said, there is still some more to it. First of all, generating large primes is not trivial, and requires the use of probabilistic primality tests, like Miller-Rabin. Similarly, the Extended Euclidean Algorithm is rather complex, at least conceptually. There are also many possible optimizations, like replacing the totient function with the reduced totient function, for which Euler's Theorem also holds. Finally, it is very important to know that you must be very mindful about security with RSA, as it is extremely easy to shoot yourself in the foot, making the entire algorithm unsecure. If you are not an expert, do not ever use a home-baked implementation of RSA for anything that actually needs to be secure. That said, understanding and implementing RSA is a fun and relatively easy project, and trying to improve its speed an security is a great learning experience in math, programming, and cryptography.
